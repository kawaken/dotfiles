# 作業開始ガイドスキル

## Description

Claude Code のベストプラクティスに基づいて、これから始める作業の進め方を案内する。
タスクの種類に応じた「何をすべきか」「Claudeにどう伝えるか」のチェックリストを提示する。

TRIGGER when:
- `/kickoff`、"作業開始"、"どう始めるか"、"ガイド"
- 新しいセッションでいきなり複数ゴール（〜して、〜も、〜あと）が含まれるタスクを受け取ったとき
- 「〜を実装して」「〜をリファクタして」など Medium/Large と判断できるタスクで、制約・完了条件・参照実装が未指定のとき

## User Invocable

yes

## Instructions

以下の手順で案内すること。

### ステップ0: セッションのリネーム

タスク内容が分かった時点で、セッション名をタスクを表す短い名前に変更するよう促すこと。

```
/rename <タスクを表す短い名前>
```

例: `/rename add-permission-check`、`/rename fix-login-bug`、`/rename refactor-auth`

**重要**: `/rename` を提示したら、そこで一旦出力を止めてユーザーの応答を待つこと。リネーム完了後またはスキップの意思表示を受けてから、ステップ0.5以降に進む。

---

### ステップ0.5: 開発環境セットアップ確認（worktree環境のみ）

**条件**: 以下の手順でworktree環境かどうかを判定し、該当する場合のみ実行する。

#### worktree判定

以下のコマンドで判定する:

```bash
git rev-parse --show-toplevel      # 現在のworktreeのルート
git rev-parse --git-common-dir     # 共通の.gitディレクトリ
```

- `--git-common-dir` の結果が `--show-toplevel`/.git と異なる（例: `../.git/worktrees/xxx`）場合はworktree環境

#### セットアップ確認手順

worktree環境の場合、以下を確認する:

| 依存管理ファイル | 対応する依存ディレクトリ | インストールコマンド |
|---|---|---|
| `package.json` | `node_modules/` | `yarn install` または `npm install` |
| `Gemfile` | `vendor/bundle/` または `Gemfile.lock` | `bundle install` |
| `go.mod` | `vendor/` | `go mod download` |
| `requirements.txt` / `pyproject.toml` | `.venv/` | `pip install -r requirements.txt` 等 |

依存管理ファイルが存在するのに対応する依存ディレクトリが存在しない場合、必要なコマンドをユーザーに提示して実行を促す。

**重要**:
- Claudeは依存インストールを勝手に実行しない
- セットアップが必要な場合はコマンドを提示し、ユーザーの応答を待ってからステップ1に進む
- セットアップ不要の場合（依存管理ファイルなし・依存ディレクトリ存在済み）は何も表示せずそのままステップ1に進む

---

### ステップ1: タスク規模の確認

引数（`$ARGUMENTS`）がある場合はそれを参照する。なければ確認する。

タスクを以下のどれかに分類する:

- **Small**（1ファイル・数行・明確な変更）
- **Medium**（複数ファイル・設計判断あり・テスト要）
- **Large**（アーキテクチャ変更・複数セッション想定）

### ステップ2: タスク種別の判定

以下のどれかに分類する（複数可）:

- **実装**: 新機能・バグ修正・リファクタ
- **調査/探索**: コードベース理解・原因調査
- **レビュー**: PR・コードの評価
- **自動化**: スクリプト・CI・hook

### ステップ3: チェックリスト提示

分類に応じて以下から該当するものを提示する。

---

#### ユーザー側チェック（自分が準備すること）

**Small タスクの場合:**
- [ ] 変更対象ファイルを特定できているか
- [ ] 完了条件を1文で言えるか
- → 「〇〇ファイルの△△を□□に変える」のように直接指示してOK

**Medium タスクの場合:**
- [ ] 参照すべき既存の実装パターンを把握しているか（あれば `@ファイル名` で渡す）
- [ ] 完了の定義を伝える（テスト通過？ビルドが通る？）
- [ ] 制約を明示する（使わないライブラリ、触らないファイル等）
- → Planモードで始めて、プランを確認してから実装させる

**Large タスクの場合:**
- [ ] 最初にClaudeにインタビューさせる
  ```
  [概要]を実装したい。AskUserQuestionで詳細をインタビューして、
  完了したらSPEC.mdに仕様をまとめて。
  ```
- [ ] インタビュー後、新しいセッションで実装開始（コンテキストをクリーンに保つ）
- [ ] 検証方法を決めておく（テストスイート・スクリーンショット・コマンド）

---

#### Claudeへの指示テンプレート

**実装タスク（Medium以上）のプロンプト構造:**
```
[何を作るか / 何を直すか]

制約:
- [使わないもの、触らないもの]
- [従うべきパターン: @参照ファイル]

完了条件:
- [テストが通る / ビルドが成功 / 等]
```

**調査タスクのプロンプト構造:**
```
[何を知りたいか] について調査して。

調査範囲: [src/auth/ など絞る]
知りたいこと: [具体的に3点以内]
```
→ 広い調査はサブエージェントに委ねる: `"サブエージェントを使って〇〇を調査して"`

---

#### セッション中のルール（Claudeへの依頼）

- **2回連続で同じ間違いをしたら**: `/clear` してより具体的なプロンプトで再開
- **コンテキストが重くなってきたら**: `/clear` でリセット（無関係な会話をリセット）
- **長い調査が必要**: `"サブエージェントを使って"` と前置きする
- **Planモードの使い所**:
  - 複数ファイルの変更 → Planモード使う
  - 1ファイル・明確な変更 → 直接指示でOK
  - 不慣れなコードベース → Planモード使う

---

#### よくあるNG指示パターン

| NG | 改善例 |
|---|---|
| "バグを直して" | "ログインが〇〇の条件で失敗する。src/auth/を確認して原因を特定し、失敗するテストを書いてから直して" |
| "ダッシュボードを良くして" | "[スクショ貼付] この実装を[デザイン貼付]に近づけて。完了したらスクショを撮って比較して" |
| "テストを追加して" | "foo.pyのユーザーがログアウトしているケースのテストを追加して。モックは使わない" |
| "調査して"（範囲広い） | "src/auth/のトークンリフレッシュ周りだけを調査して、セッション管理の流れをまとめて" |

---

### ステップ4: 「最初の一言」のドラフト提示

タスク内容が分かっていれば、Claudeに渡す最初のプロンプトのドラフトを提示する。
ユーザーが加筆修正して使えるように、`コードブロック` で出力する。
