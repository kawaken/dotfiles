# gtgt（グツグツ）

## Description

プランを反復的に煮詰めるスキル。グツグツと議論を重ねながら、考慮漏れや前提の甘さを炙り出し、計画を凝縮・強化する。
Devil's Advocate（悪魔の代弁者）で前提を揺さぶり、体系的分析で技術的実現可能性と要件網羅性を検証し、構造化された質問で意思決定を固める。

TRIGGER when: user invokes `/gtgt`, "プランを煮詰めて", "計画を見直して", "グツグツして", "rethink this plan"

## User Invocable

yes

## Arguments

- plan: (任意) 煮詰めるPlanファイルのパス。省略時は会話コンテキストから推定する。
- focus: (任意) 煮詰めの焦点。例: "データ整合性", "パフォーマンス"。指定時は Phase 1-2 でその観点を重点的に分析し、他の観点は致命的な問題のみ報告する。

## Instructions

以下の3フェーズを一括で自動実行し、プランを煮詰める。
Phase 1-2の分析結果はそのまま表示し、Phase 3でチェックリスト形式のサマリーを提示して初めてユーザーに確認を取る。

---

### 前準備

1. `plan` 引数が指定されていればそのファイルを Read で読む
2. 指定がない場合、直前の会話コンテキストからプランの内容を把握する
3. プランで言及されているコードベースのファイルを Glob/Grep/Read で確認する
4. 以下を整理する:
   - 変更対象ファイル一覧
   - 依存関係
   - 想定される影響範囲
5. `focus` 引数が指定されている場合、Phase 1-2 でその観点を重点的に分析する。他の観点は致命的な問題のみ報告する。

---

### Phase 1: Devil's Advocate（悪魔の代弁者）

意図的に批判的な視点でプランに挑む。以下の観点で分析する:

- **前提の検証**: プランが暗黙的に正しいとしていることは何か？それが崩れたら何が起きるか？
- **最悪のシナリオ**: この実装で起こりうる最悪の事態は何か？
- **論理的矛盾・ギャップ**: ステップ間に矛盾や依存の循環はないか？
- **見落とし**: 既存コードとの衝突、考慮されていない副作用はないか？

結果を以下のフォーマットで提示する:

```
## Phase 1: Devil's Advocate の所見

### 危険な前提
- [前提]: なぜ危険か → こうなる可能性

### 最悪のシナリオ
- [シナリオ]: 発生条件 → 影響

### 論理的矛盾・ギャップ
- [矛盾点]: 詳細
```

（確認なしでPhase 2に自動で進む）

---

### Phase 2: Systematic Analysis（体系的分析）

2軸で構造化された分析を行う。Glob/Grep/Read で実際のコードを確認しながら実施する。

**技術的実現可能性:**
- 既存コードとの互換性
- 依存関係の整合性
- パフォーマンスへの影響
- 既存パターンとの一貫性（類似実装が既にないか確認。Grep で類似機能のコードを検索し、DRY原則・エラーハンドリングパターン・命名規約との整合性も合わせて検証する）

**要件網羅性:**
- エッジケースの洗い出し
- エラーハンドリングの網羅
- 境界条件の確認
- 未考慮シナリオの特定

結果をテーブル形式で提示する:

```
## Phase 2: 体系的分析の所見

### 技術的実現可能性
| 観点 | 状態 | 詳細 |
|------|------|------|
| 既存コード互換性 | ✅/⚠️/❌ | ... |
| 依存関係 | ✅/⚠️/❌ | ... |
| パフォーマンス | ✅/⚠️/❌ | ... |
| パターン一貫性 | ✅/⚠️/❌ | ... |

### 要件網羅性
| 観点 | 状態 | 詳細 |
|------|------|------|
| エッジケース | ✅/⚠️/❌ | ... |
| エラーハンドリング | ✅/⚠️/❌ | ... |
| 境界条件 | ✅/⚠️/❌ | ... |
| 未考慮シナリオ | ✅/⚠️/❌ | ... |
```

（確認なしでPhase 3に自動で進む）

---

### Phase 3: チェックリスト形式サマリー

Phase 1・2 の所見をインパクト順に統合し、Yes/No チェックリスト形式で提示する。
指摘事項がない場合は「特に指摘事項なし」と明示して完了サマリーに進む。

**チェックリストのルール:**
- 項目数は最大8つまで（認知負荷を抑えるため）
- 各項目は「〜は問題ない？」「〜で合ってる？」のようなYes/No形式
- カテゴリ別にグルーピングする（前提・リスク / 技術的実現可能性 / 要件網羅性）
- 問題なしと判断できる項目は含めない（指摘事項のみ）

**提示方法:**
1. テキストで「以下の指摘事項を確認してな。気になる項目を選んでな。選ばんかった項目は問題なしとして扱うで。」と前置きする
2. AskUserQuestion（multiSelect: true）で指摘事項を選択肢として提示する
3. ユーザーは気になる項目を複数選択できる（選択された項目 = 問題ありとして深掘り対象）

**ユーザー回答後の処理:**
1. 選択された（問題あり）項目についてのみ深掘りし、対応策を提示
2. プランファイルを Write/Edit で直接更新する
3. 変更サマリーを提示する

---

### 反復判定

チェックリストへの回答後、以下を評価する:

- 選択された問題はすべて深掘り・対応済みか？
- 新たな対応によって新しい問題が生まれていないか？

**重大な未解決事項がある場合**: 修正を反映した上で、再度チェックリストを提示する。

**すべて解決済みの場合（指摘事項なし含む）**: 最終サマリーを提示して完了:

```
## グツグツ完了サマリー

### 実施したレビューラウンド数: N回

### 主要な改善点
1. [改善点1]
2. [改善点2]
...

### Planファイルへの変更箇所
- [変更1]
- [変更2]

プランが煮詰まったで。実装に進んでええ！
```

**反復の上限**: 最大3ラウンドを目安とする。それ以上続く場合はユーザーにその旨を伝えて判断を仰ぐ。
