# rethink

## Description

プランの反復的レビュー・改善を行うスキル。
Devil's Advocate（悪魔の代弁者）アプローチでプランの前提を揺さぶり、体系的分析で技術的実現可能性と要件網羅性を検証し、構造化された質問でプランを磨き上げる。

TRIGGER when: user invokes `/rethink`, "プランを見直して", "計画を再検討して", "rethink this plan"

## User Invocable

yes

## Arguments

- plan: (任意) レビュー対象のPlanファイルパス。省略時は会話コンテキストから推定する。

## Instructions

以下の3フェーズを反復的に実行し、プランを改善する。
各フェーズの終わりにユーザーに確認してから次へ進む。

---

### 前準備

1. `plan` 引数が指定されていればそのファイルを Read で読む
2. 指定がない場合、直前の会話コンテキストからプランの内容を把握する
3. プランで言及されているコードベースのファイルを Glob/Grep/Read で確認する
4. 以下を整理する:
   - 変更対象ファイル一覧
   - 依存関係
   - 想定される影響範囲

---

### Phase 1: Devil's Advocate（悪魔の代弁者）

意図的に批判的な視点でプランに挑む。以下の観点で分析する:

- **前提の検証**: プランが暗黙的に正しいとしていることは何か？それが崩れたら何が起きるか？
- **最悪のシナリオ**: この実装で起こりうる最悪の事態は何か？
- **論理的矛盾・ギャップ**: ステップ間に矛盾や依存の循環はないか？
- **見落とし**: 既存コードとの衝突、考慮されていない副作用はないか？

結果を以下のフォーマットで提示する:

```
## Phase 1: Devil's Advocate の所見

### 危険な前提
- [前提]: なぜ危険か → こうなる可能性

### 最悪のシナリオ
- [シナリオ]: 発生条件 → 影響

### 論理的矛盾・ギャップ
- [矛盾点]: 詳細
```

提示後、「Phase 2 に進んでええ？」とユーザーに確認する。

---

### Phase 2: Systematic Analysis（体系的分析）

2軸で構造化された分析を行う。Glob/Grep/Read で実際のコードを確認しながら実施する。

**技術的実現可能性:**
- 既存コードとの互換性
- 依存関係の整合性
- パフォーマンスへの影響
- 既存パターンとの一貫性（類似実装が既にないか確認）

**要件網羅性:**
- エッジケースの洗い出し
- エラーハンドリングの網羅
- 境界条件の確認
- 未考慮シナリオの特定

結果をテーブル形式で提示する:

```
## Phase 2: 体系的分析の所見

### 技術的実現可能性
| 観点 | 状態 | 詳細 |
|------|------|------|
| 既存コード互換性 | ✅/⚠️/❌ | ... |
| 依存関係 | ✅/⚠️/❌ | ... |
| パフォーマンス | ✅/⚠️/❌ | ... |
| パターン一貫性 | ✅/⚠️/❌ | ... |

### 要件網羅性
| 観点 | 状態 | 詳細 |
|------|------|------|
| エッジケース | ✅/⚠️/❌ | ... |
| エラーハンドリング | ✅/⚠️/❌ | ... |
| 境界条件 | ✅/⚠️/❌ | ... |
| 未考慮シナリオ | ✅/⚠️/❌ | ... |
```

提示後、「Phase 3 に進んでええ？」とユーザーに確認する。

---

### Phase 3: 構造化質問と意思決定

Phase 1・2 の所見からインパクトの大きい問題を優先し、AskUserQuestion で選択肢付き質問を提示する。

**質問のルール:**
- 一度に最大3つまで（認知負荷を抑えるため）
- 各質問に2〜4個の具体的な選択肢
- 各選択肢には pros/cons を簡潔に含める
- 曖昧な質問は避け、選択肢を具体的に

**回答後の処理:**

1. プランファイルを Write/Edit で直接更新する
2. 変更内容をサマリーとして提示する:

```
## 意思決定の反映サマリー

| 質問 | 決定 | プランへの変更内容 |
|------|------|--------------------|
| Q1 | [選択] | [具体的な変更] |
| Q2 | [選択] | [具体的な変更] |
```

---

### 反復判定

Phase 3 完了後、以下を評価する:

- Phase 1〜3 で発見された問題はすべて解決されたか？
- 新たな意思決定によって新しい問題が生まれていないか？

**重大な未解決事項がある場合**: 「まだ検討すべき点があるから、もう1周回してええ？」とユーザーに確認し、Phase 1 に戻る。

**すべて解決済みの場合**: 最終サマリーを提示して完了:

```
## Rethink 完了サマリー

### 実施したレビューラウンド数: N回

### 主要な改善点
1. [改善点1]
2. [改善点2]
...

### Planファイルへの変更箇所
- [変更1]
- [変更2]

プランの改善完了したで。実装に進んでええ！
```

**反復の上限**: 最大3ラウンドを目安とする。それ以上続く場合はユーザーにその旨を伝えて判断を仰ぐ。
