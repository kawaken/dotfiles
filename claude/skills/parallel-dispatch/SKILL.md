# parallel-dispatch

## Description

複数の独立したタスクを並列サブエージェントに委任するスキル。
共有状態なし・順序依存なしの条件を満たす複数タスクを検出したら、自動で並列ディスパッチして効率化する。
完了後に何を並列で実行したか・なぜ並列にできたかをInsightsとして報告する。

TRIGGER when: 2つ以上の独立したタスク（共有状態なし・順序依存なし）に直面した時、またはユーザーが明示的に並列実行を指示した時

## User Invocable

yes

### トリガー例
- `/parallel`、"並列で"、"パラレルで"
- "並列にやって"、"同時に進めて"

## Instructions

### 独立性チェック（使う前の判断）

以下をすべて満たす場合のみ並列ディスパッチを使う:

- [ ] タスク間で共有するファイル・状態がない
- [ ] タスクAの結果をタスクBが必要としない（順序依存なし）
- [ ] 各タスクが独立して検証可能

**使わない場面**（直列で対応する）:
- 関連する障害のデバッグ（原因が連鎖している可能性）
- 共有コンポーネントを同時編集
- 探索的な調査（何を調べるか自体が不明）
- 最初のタスクの結果によって次のアプローチが変わる

---

### よくある並列パターン

状況に応じて以下のパターンを活用する:

#### レビュー分割
PRレビューをロール別に並列化する:
```
Track 1: バックエンドレビュー（ロジック・権限・DB設計）
Track 2: フロントエンドレビュー（UI・型・アクセシビリティ）
Track 3: テスト検証（既存テスト実行・カバレッジ確認）
```
→ 各Trackは担当範囲のファイルのみを参照する。言語・フレームワークはプロジェクトに応じて判断。

#### 調査分割
独立したコンポーネント・モジュールの調査を並列化する:
```
Agent A: [コンポーネントAの調査]
Agent B: [コンポーネントBの調査]
```
→ 調査対象に依存関係がない場合のみ並列にする。

#### 実装分割
互いに独立したファイル群への変更を並列化する:
```
Agent A: [ファイル群Aへの変更]
Agent B: [ファイル群Bへの変更]
```
→ 同じファイルを複数エージェントが編集しないよう制約を明示する。

---

### 並列ディスパッチの実行

#### エージェントプロンプトの構造

各サブエージェントへの指示は以下の4要素で構成する:

```
**スコープ**: [このエージェントが担当する範囲]
**ゴール**: [達成すべき具体的な成果物]
**制約**: [触れてはいけないファイル・範囲、従うべき規約]
**期待出力**: [返してほしい情報の形式]
```

#### 実行方法

Agent toolを使い、独立したタスクを一つのメッセージで複数同時に呼び出す。
各エージェントには上記4要素を含むプロンプトを渡す。

---

### 統合フロー

1. 各エージェントのサマリーを確認
2. 競合チェック（同じファイルを別々に変更していないか）
3. 必要に応じてフルテスト実行

---

### 並列ディスパッチ Insights（完了後に必ず報告）

実行後、以下のフォーマットでユーザーに報告する:

```
## 並列ディスパッチ Insights

### 並列にしたタスク
- タスクA: [概要]
- タスクB: [概要]

### なぜ並列にできたか
[具体的な理由: 例「この2つはそれぞれ別のファイルを対象にしてて、共有状態がないから並列にしたで」]

### 結果
- タスクA: [完了/エラー + 概要]
- タスクB: [完了/エラー + 概要]
```

このInsightsはユーザーが「ここは並列にできる」と自分で判断できるようになるためのヒントとして機能する。
